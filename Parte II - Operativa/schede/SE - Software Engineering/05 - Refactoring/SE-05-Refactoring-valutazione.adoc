= SE-05 - Refactoring - Valutazione

== L0

Non riconosce problemi significativi o propone interventi che cambiano il comportamento senza protezioni.

* Domanda per L0: da dove partiresti per non introdurre regressioni?
* Domanda per L0: quale parte del comportamento vorresti verificare prima di toccare il codice?

== L1

Riconosce smell evidenti nel codice, ma in modo frammentario e senza un primo passo sicuro.

* [ ] Nota nomi deboli, duplicazioni o condizioni ripetute
* [ ] Identifica almeno un punto di semplificazione locale
* [ ] Distingue in modo parziale tra miglioramento e cambio funzionale
* [ ] Parla di test solo in modo generico
* Domanda per L1: quale sarebbe il cambiamento più piccolo e meno rischioso?
* Domanda per L1: quali segnali ti fanno dire che il codice va protetto prima di modificarlo?

== L2

Propone una strategia di base coerente e orientata a piccoli passi verificabili.

* [ ] Parte da test di caratterizzazione o dai test già presenti
* [ ] Propone un refactoring locale, per esempio estrazione di metodi
* [ ] Separa almeno una responsabilità evidente
* [ ] Mantiene esplicito il controllo del comportamento esistente
* [ ] Riconosce casi utili da coprire: righe con e senza sconto, cliente VIP, paese, arrotondamento
* Domanda per L2: quale test aggiungeresti per primo e perché?
* Domanda per L2: quale responsabilità estrarresti prima dal metodo?

== L3

Motiva le scelte, tratta casi non banali e collega il refactoring alla protezione del dominio.

* [ ] Spiega perché certi passaggi riducono il rischio di regressione
* [ ] Considera l'interazione tra sconto di riga, sconto VIP e tasse
* [ ] Decide l'ordine dei passi in modo coerente con i test
* [ ] Riconosce dove fermarsi nella prima iterazione
* [ ] Distingue chiaramente refactoring da modifica funzionale
* Domanda per L3: come verificheresti che il comportamento sia rimasto invariato dopo ogni passo?
* Domanda per L3: quali casi aggiungeresti prima di toccare la logica VIP o fiscale?

== L4

Ragiona su evoluzione, compatibilità, migrazione e impatti sul design futuro.

* [ ] Valuta l'effetto delle nuove regole sul codice già scritto
* [ ] Considera compatibilità con comportamenti esistenti e casi limite
* [ ] Propone una struttura che faciliti estensioni future
* [ ] Anticipa rischi di accoppiamento tra regole diverse
* [ ] Distingue il refactoring immediato da un'evoluzione più ampia
* Domanda per L4: come prepareresti il codice per nuove regole fiscali senza riscriverlo tutto?
* Domanda per L4: quale compromesso faresti tra semplicità immediata ed estensibilità futura?

== L5

Guida una scelta completa, con trade-off espliciti, criteri decisionali e prevenzione dei problemi futuri.

* [ ] Definisce una strategia di refactoring sostenibile per il team
* [ ] Seleziona criteri chiari per fermarsi o continuare
* [ ] Bilancia test, leggibilità, rischio e manutenibilità
* [ ] Propone come prevenire regressioni nelle evoluzioni successive
* [ ] Riesce a giustificare alternative diverse senza perdere coerenza
